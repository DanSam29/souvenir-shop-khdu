Примітки до валідації:
•	POST /api/warehouse/companies: перевіряє унікальність name та email
•	PUT /api/warehouse/companies/{id}: перевіряє унікальність email якщо змінено
•	POST /api/warehouse/companies/{id}/deactivate: перевіряє чи фірма не використовується в активних документах
•	POST /api/warehouse/incoming: перевіряє що companyId відповідає активній фірмі (isActive=true)
•	POST /api/warehouse/outgoing: якщо reason=RETURN, то companyId обов'язковий та перевіряється на активність фірми

Примітки:
Права доступу:
•	Менеджер (MANAGER) має доступ до:
o	/api/analytics/sales
o	/api/analytics/products/popular
o	/api/analytics/categories
o	/api/analytics/promotions/effectiveness
o	/api/analytics/export (тільки для доступних звітів)
•	Адміністратор (ADMIN) має доступ до всіх endpoints, включаючи:
o	/api/analytics/conversion
o	/api/analytics/financial
o	/api/analytics/user-behavior
Валідація запитів:
•	startDate та endDate: обов'язкові для більшості endpoints
•	Період не може перевищувати 2 роки (730 днів)
•	groupBy: має бути один з ["day", "week", "month"], default: "day"
•	metric: має бути один з ["quantity", "revenue", "profit"]
•	format: має бути один з ["PDF", "CSV", "Excel"]
•	limit: має бути в діапазоні 1-100, default: 10
Кешування:
•	Всі розраховані метрики кешуються в In-Memory Cache
•	TTL для операційних звітів (sales, products, categories): 15 хвилин
•	TTL для фінансових звітів: 30 хвилин
•	TTL для метрик конверсії: 1 година
•	Ключ кешу: "analytics:{type}:{period}:{filters_hash}"
Обмеження продуктивності:
•	Rate limiting: 10 запитів/хвилину на користувача для аналітики
•	Обмеження на експорт: 5 експортів/годину на користувача
•	Асинхронна генерація звітів для датасетів > 5000 записів
Коди помилок:
•	400 Bad Request: невалідні параметри (період > 2 роки, невалідний groupBy)
•	403 Forbidden: недостатньо прав (користувач намагається отримати financial/conversion/user-behavior)
•	429 Too Many Requests: перевищено rate limit
•	500 Internal Server Error: помилка при розрахунку метрик або генерації звіту

Примітки до PromotionModule
Стакування знижок:
Система підтримує стакування знижок за пріоритетом:
1.	Персональні знижки (CUSTOM) - найвищий пріоритет
2.	Студентські знижки (STUDENTS) - залежно від studentStatus
3.	Товарні/Категорійні знижки (PRODUCT/CATEGORY)
4.	Знижки на кошик (CART)
5.	Промокоди
При однаковому типі знижок застосовується поле priority (вищий = важливіший).
Розрахунок фінальної ціни:
finalPrice = originalPrice - discountAmount
де discountAmount розраховується на основі типу знижки:
•	PERCENTAGE: originalPrice * (value / 100)
•	FIXED_AMOUNT: value
•	SPECIAL_PRICE: originalPrice - value
Валідація промокоду:
Перевірки виконуються в наступному порядку:
1.	Існування промокоду в БД
2.	isActive = true
3.	startDate <= now() <= endDate (якщо вказано)
4.	currentUsage < usageLimit (якщо вказано)
5.	orderTotal >= minOrderAmount (якщо вказано)
6.	totalQuantity >= minQuantity (якщо вказано)
При невдалій валідації система повертає конкретне повідомлення про причину.
Примітки до UniversityModule
Автоматична верифікація:
При реєстрації користувача з email що закінчується на @ksu.edu.ua або @student.ksu.edu.ua система автоматично намагається верифікувати статус студента через University API.
Мапування studentStatus:
На основі відповіді University API:
•	GPA >= 4.5 AND scholarshipStatus = "ACTIVE" → HIGH_ACHIEVER
•	scholarshipStatus = "ACTIVE" → SCHOLARSHIP
•	інакше → REGULAR
Обробка помилок API:
Якщо University API недоступний або повертає помилку:
•	Система логує помилку
•	Створює запис в StudentVerificationQueue
•	Встановлює nextRetryAt = now() + 1 година
•	Користувач отримує studentStatus = NONE (тимчасово)
•	Повторні спроби: через 1 год, 6 год, 24 год
•	Після 3 невдалих спроб → переводиться в ручну верифікацію адміністратором
Термін дії studentStatus:
За замовчуванням 4 місяці (1 семестр). За тиждень до закінчення терміну користувач отримує Email-нагадування з пропозицією оновити статус. Адміністратор може продовжити термін дії вручну.
Безпека:
University API викликається ТІЛЬКИ через backend. Frontend ніколи не має прямого доступу до University API. API ключ зберігається в environment variables та не передається клієнту.
Примітки до AnalyticsModule
Стратегія розрахунку метрик:
AnalyticsModule використовує on-the-fly розрахунки з агресивним кешуванням:
1.	При отриманні запиту формується ключ кешу на основі параметрів
2.	Перевіряється наявність даних у кеші
3.	Якщо дані актуальні (TTL не вийшов) → повертається з кешу
4.	Якщо кеш порожній або застарілий:
•	Запитуються дані з репозиторіїв
•	Метрики розраховуються паралельно (Task.WhenAll)
•	Результат зберігається в кеш
•	Повертається клієнту
Паралельний розрахунок метрик:
	Для прискорення розрахунку складних звітів використовується паралельна обробка:
•	Aggregate metrics (totalOrders, totalRevenue) - Task 1
•	Time series (salesByDay/Week/Month) - Task 2
•	Distributions (salesByPaymentMethod, salesByStatus) - Task 3
•	Top lists (topProducts, topCategories) - Task 4
•	Comparisons (changeVsPreviousPeriod) - Task 5
Всі задачі виконуються одночасно, результати об'єднуються в фінальний DTO.
Інвалідація кешу:
	Кеш аналітики автоматично інвалідується при:
•	Створенні нового замовлення (Order)
•	Зміні статусу замовлення (Order.status)
•	Створенні нової прибуткової накладної (IncomingDocument)
•	Створенні нової видаткової накладної (OutgoingDocument)
•	Активації/деактивації акції (Promotion.isActive)
Інвалідація виконується через pub/sub механізм або через зміну версії кешу.
	Асинхронна генерація великих звітів:
		Якщо датасет для звіту > 5000 записів:
•	Користувач отримує HTTP 202 Accepted з jobId
•	Звіт генерується асинхронно в background job
•	Користувач може перевіряти статус через GET /api/analytics/export/{jobId}/status
•	Після завершення генерації доступний download link
•	Згенеровані файли зберігаються 24 години, потім автоматично видаляються
Формати експорту:
•	PDF:
o	Використання iText7 для генерації
o	Включення графіків як зображень (Chart.js → Canvas → PNG → PDF)
o	Форматування: титульна сторінка, KPI картки, графіки, таблиці, footer
o	Розмір файлу: зазвичай 500KB - 2MB
•	Excel (XLSX):
o	Використання EPPlus для генерації
o	Декілька листів: Summary, Sales Dynamics, Top Products, Charts
o	Форматування комірок (числа, дати, валюта)
o	Автоматична ширина колонок, freeze panes
o	Вбудовані Excel charts
o	Розмір файлу: зазвичай 200KB - 1MB
•	CSV:
o	Використання CsvHelper для генерації
o	UTF-8 encoding з BOM (для коректного відкриття в Excel)
o	Comma separator (або semicolon для європейських локалей)
o	Тільки табличні дані, без графіків
o	Ідеально для подальшої обробки в BI tools
o	Розмір файлу: зазвичай 50KB - 500KB
Безпека та аудит:
•	Всі запити до аналітики логуються:
o	Хто (userId, role)
o	Що (тип звіту, параметри)
o	Коли (timestamp)
o	Результат (success/error, execution time)
