Вимоги інтернет-магазину з продажу сувенірної продукції Херсонського державного університету
1. Вступ
Даний документ містить вимоги до інформаційної системи інтернет-магазину сувенірної продукції Херсонського державного університету. Система покликана забезпечити можливість студентам, працівникам та гостям університету зручно замовляти та купувати фірмові товари навчального закладу онлайн. Документ визначає цілі, завдання, функціональні та нефункціональні вимоги, що слугуватимуть основою для подальшого етапу проєктування та реалізації системи.
2. Мета та завдання системи
2.1. Мета проєкту:
Метою проєкту є створення веб-додатку інтернет-магазину сувенірної продукції Херсонського державного університету, який забезпечить зручний, безпечний та автоматизований процес продажу університетських товарів онлайн. Система має забезпечити повний цикл електронної комерції - від перегляду асортименту до оплати, доставки та управління товарами адміністрацією.
2.2. Завдання проєкту:
1. Реалізація інтуїтивно зрозумілого інтерфейсу для користувачів із можливістю перегляду, пошуку, фільтрації та замовлення товарів.
2. Забезпечення можливості створення користувацьких акаунтів, ведення історії замовлень і управління профілем.
3. Реалізація системи кошика та оформлення замовлення з інтеграцією платіжних сервісів.
4. Автоматизація обробки замовлень, відображення статусів оплати та доставки.
5. Інтеграція із зовнішніми сервісами доставки (Nova Poshta API) для відстеження посилок і вибору пункту доставки.
6. Розробка адміністративної панелі для керування товарами, користувачами, замовленнями та фінансовими операціями.
7. Забезпечення інформаційної безпеки користувацьких даних і фінансових транзакцій.
8. Формування звітів щодо продажів, фінансових потоків і статистики користування системою.
9. Створення адаптивного дизайну для коректного відображення на ПК, планшетах і смартфонах.
10. Реалізація складського обліку через прибуткові та видаткові накладні для документування руху товарів.
11. Реалізація повнофункціональної системи акцій та знижок з підтримкою: 
•	Різних типів знижок (відсоткові, фіксовані, спеціальна ціна) 
•	Цільових аудиторій (всі користувачі, студенти, персональні) 
•	Промокодів для активації знижок 
•	Автоматичної верифікації студентів через University API 
•	Стакування знижок за пріоритетом 
•	Аналітики ефективності акцій
12. Побудова архітектури, яка дозволить подальше розширення функціоналу (наприклад, програма лояльності, бонуси, реферальна система).
2.3. Цільова аудиторія:
•	Студенти ХДУ: як основні покупці університетських сувенірів.
•	Викладачі та працівники університету: для придбання продукції та корпоративних товарів.
•	Випускники університету: для підтримки зв’язку з alma mater через придбання символічної атрибутики.
•	Гості університету, відвідувачі заходів: як потенційні покупці сувенірної продукції.
•	Партнери та представники інших навчальних закладів: для корпоративних замовлень або обміну сувенірною продукцією.
•	Адміністрація університету: для керування асортиментом, обліком замовлень і фінансами.
2.4. Ролі користувачів:
1. Гість
Користувач, який не авторизований в системі. Має обмежений доступ для перегляду каталогу та пошуку.
2. Користувач
Зареєстрований клієнт магазину. Має доступ до особистого кабінету, історії замовлень та можливість оформлення покупок.
3. Менеджер
Співробітник, відповідальний за наповнення сайту. Керує каталогом товарів, акціями та знижками.
4. Адміністратор
Користувач з повним доступом до управління замовленнями, користувачами та фінансовими звітами.
5. СуперАдмін
Технічний адміністратор з правами на управління системою, ролями та інтеграціями.

3.2. Функціональні вимоги для Користувача
FR-C-01: Додавання товарів до кошика
Опис: Користувач може додавати товари до кошика для подальшого оформлення замовлення.
Ризик: Середній (Ризик втрати даних кошика при оновленні сторінки або некоректного врахування товарів.)
Пріоритет: Високий
FR-C-02: Оформлення замовлення
Опис: Користувач може оформляти замовлення з вибором населеного пункту, відділення доставки Nova Poshta та способу оплати. Система підтримує два способи оплати: онлайн-оплата через платіжний шлюз Stripe та оплата при отриманні на відділенні Nova Poshta. Вартість доставки розраховується автоматично через Nova Poshta API на основі ваги товарів та обраного відділення. При оформленні замовлення система автоматично: 
•	Перевіряє всі доступні знижки для користувача (студентські, персональні, акційні) 
•	Застосовує найвигідніші знижки за пріоритетом 
•	Дозволяє ввести промокод для додаткової знижки 
•	Застосовує стакування знижок (автоматичні + промокод) 
•	Розраховує фінальні ціни з урахуванням всіх знижок 
•	Зберігає повну інформацію про ціни та знижки в OrderItems (originalPrice, appliedPromotionId, discountAmount, finalPrice) 
При успішному оформленні замовлення автоматично створюється видаткова накладна через WarehouseModule для резервування товарів на складі з копіюванням інформації про знижки з OrderItems. Після успішного оформлення користувач отримує Email-підтвердження з детальною деталізацією: вартість товарів без знижок, список застосованих знижок, вартість доставки, підсумкова сума.
Ризик: Високий (Ризик втрати даних замовлення, некоректного розрахунку вартості доставки та знижок, помилок при інтеграції з платіжними сервісами, Nova Poshta API або University API, недоступність зовнішніх сервісів, некоректне стакування знижок.)
Пріоритет: Високий
FR-C-03: Керування особистим кабінетом
Опис: Користувач може редагувати свої контактні дані, змінювати пароль, керувати адресами доставки.
Ризик: Низький (Ризик некоректного оновлення даних або порушення конфіденційності.)
Пріоритет: Середній
FR-C-04: Перегляд історії замовлень
Опис: Користувач може переглядати повну історію своїх замовлень з деталізацією.
Ризик: Низький (Ризик відображення неповної або некоректної інформації.)
Пріоритет: Середній
FR-C-05: Відстеження статусів замовлення
Опис: Користувач може відстежувати поточний статус свого замовлення (Обробляється, Відправлено, Доставлено) в особистому кабінеті, через Email-сопвіщення.
Ризик: Середній (Ризик несвоєчасного оновлення статусу, що призведе до плутанини.)
Пріоритет: Середній
FR-C-06: Зберігання даних для доставки
Опис: Користувач може зберігати кілька адрес доставки для швидкого оформлення замовлень. Кожна збережена адреса включає: населений пункт, відділення Nova Poshta, контактний телефон. Це дозволяє швидко обирати збережену адресу при повторному оформленні замовлення.
Ризик: Низький (Ризик некоректного збереження або відображення адрес.)
Пріоритет: Низький
FR-C-07: Отримання персональних пропозицій та автоматичних знижок 
Опис: Користувач може отримувати персональні рекомендації товарів на основі історії переглядів і покупок. Система автоматично застосовує доступні знижки на основі статусу користувача: 
•	Студентські знижки (автоматично після верифікації через University API) 
•	Персональні знижки (призначені менеджером/адміністратором) 
•	Акційні знижки (активні промо-акції на товари/категорії)
Знижки застосовуються автоматично при перегляді товарів та оформленні замовлення. Користувач отримує Email-сповіщення про нові доступні знижки.
Ризик: Середній (Ризик некоректних рекомендацій через помилки в алгоритмі, некоректного застосування знижок.)
Пріоритет: Середній
FR-C-08: Перегляд доступних знижок
Опис: Користувач може переглядати список всіх доступних йому знижок в особистому кабінеті. Відображається інформація про кожну знижку: назва, опис, розмір знижки, область застосування (товари/категорії/весь асортимент), термін дії, умови використання (мінімальна сума замовлення, мінімальна кількість товарів). Знижки групуються за типами: студентські, персональні, загальні акції.
Ризик: Низький (Ризик відображення неактуальної інформації про знижки через помилки кешування.) 
Пріоритет: Середній
FR-C-09: Верифікація студентського статусу
Опис: При реєстрації з університетським email (@ksu.edu.ua або @student.ksu.edu.ua) система автоматично відправляє запит до University API для верифікації статусу студента. При успішній верифікації система отримує дані: studentStatus (REGULAR/SCHOLARSHIP/HIGH_ACHIEVER), GPA, scholarshipStatus. На основі цих даних автоматично призначаються студентські знижки:
•	REGULAR: базова студентська знижка
•	SCHOLARSHIP (стипендіат): підвищена знижка
•	HIGH_ACHIEVER (GPA ≥ 4.5): максимальна знижка
Студентський статус має термін дії (4 місяці до кінця семестру), після чого знижки деактивуються. Користувач отримує Email-підтвердження з інформацією про присвоєні знижки. Якщо University API недоступний, система створює задачу для повторної перевірки через 1 годину. Якщо студент не знайдений в базі університету, адміністратор може верифікувати статус вручну на основі документів.
Ризик: Середній (Ризик недоступності University API, некоректної верифікації, помилок при призначенні знижок.)
Пріоритет: Високий
FR-C-10: Введення промокоду при оформленні
Опис: Користувач може ввести промокод при оформленні замовлення для отримання додаткової знижки. Система валідує промокод:
•	Перевіряє існування в БД
•	Перевіряє чи активний (isActive = true)
•	Перевіряє термін дії (startDate ≤ now ≤ endDate)
•	Перевіряє ліміт використань (currentUsage < usageLimit)
•	Перевіряє умови застосування (мінімальна сума замовлення, мінімальна кількість товарів)
При успішній валідації промокод застосовується поверх автоматичних знижок (стакування). Система перераховує фінальні ціни та відображає оновлену деталізацію замовлення. При невалідному промокоді система показує відповідне повідомлення: "Промокод не знайдено", "Термін дії закінчився", "Промокод вичерпано", "Не відповідає умовам". Після успішного оформлення замовлення система збільшує лічильник використання промокоду (currentUsage++).
Ризик: Середній (Ризик некоректної валідації промокоду, помилок при стакуванні знижок, можливість зловживання промокодами.)
Пріоритет: Середній
FR-C-11: Перегляд історії використаних знижок
Опис: Користувач може переглядати історію використання знижок в особистому кабінеті. Для кожної використаної знижки відображається: назва акції, дата використання, номер замовлення, сума знижки в грн. Історія доступна з фільтрами за датою та типом знижки.
Ризик: Низький (Ризик відображення неповної інформації через помилки в запитах до БД.)
Пріоритет: Низький
3.3. Функціональні вимоги для Менеджера
FR-M-01: Керування каталогом товарів
Опис: Менеджер може додавати, редагувати, видаляти товари в каталозі, керувати їх атрибутами (назва, опис, ціна, вага). Кількість товару на складі не вказується при створенні товару, а керується через оформлення прибуткових накладних. Поточні залишки розраховуються автоматично на основі всіх прибуткових та видаткових документів.
Ризик: Середній (Ризик некоректного введення даних, що призведе до помилок у замовленнях.)
Пріоритет: Високий
FR-M-02: Керування категоріями товарів
Опис: Менеджер може створювати, редагувати, видаляти категорії товарів та налаштовувати їх ієрархію.
Ризик: Низький (Ризик порушення структури каталогу при некоректному редагуванні.)
Пріоритет: Високий
FR-M-03: Завантаження зображень товарів
Опис: Менеджер може завантажувати, редагувати та видаляти зображення товарів через адмін-панель.
Ризик: Низький (Ризик завантаження невідповідних форматів або перевищення розмірів файлів.)
Пріоритет: Високий
FR-M-04: Керування акціями та знижками
Опис: Менеджер має повний функціонал керування системою акцій та знижок:
•	Менеджер може створити нову акцію з наступними параметрами:
o	Назва та опис акції
o	Тип знижки: PERCENTAGE (відсоткова), FIXED_AMOUNT (фіксована сума в грн), SPECIAL_PRICE (спеціальна ціна)
o	Значення знижки (число)
o	Цільова аудиторія: ALL (всі користувачі), STUDENTS (студенти), STAFF (працівники), ALUMNI (випускники), CUSTOM (персонально)
o	Область застосування: PRODUCT (конкретний товар), CATEGORY (категорія товарів), CART (весь кошик), SHIPPING (доставка)
o	Дати дії (опціонально): startDate, endDate
o	Промокод (опціонально): унікальний код для активації
o	Умови застосування (опціонально): мінімальна сума замовлення, мінімальна кількість товарів
o	Пріоритет (0-100): для визначення порядку застосування при стакуванні
o	Ліміт використань (опціонально): максимальна кількість застосувань
•	Менеджер може редагувати всі параметри існуючої акції.
•	Менеджер може деактивувати акцію (isActive = false) або повністю видалити.
•	Менеджер може призначити знижку конкретному користувачу (наприклад, VIP клієнтам).
•	Менеджер може переглядати статистику використання знижок:
o	Кількість використань кожної акції
o	Загальна сума знижок
o	Найпопулярніші акції
o	ROI по акціях (окупність)
o	Експорт звіту по акціях у форматах PDF/CSV/Excel
•	Менеджер може фільтрувати акції за статусом (активні/неактивні), типом, цільовою аудиторією, датами.
•	Менеджер може переглядати розширену аналітику ефективності акцій:
o	Всі метрики що вже є (кількість використань, загальна сума знижок, ROI)
o	Порівняння конверсії з акцією vs без акції

Система підтримує стакування знижок за пріоритетом:

Якщо потрібної фірми немає в списку, менеджер може додати нову фірму безпосередньо з форми оформлення накладної. При збереженні накладної система автоматично збільшує поточний залишок товару на складі. Всі операції логуються для аудиту.

FR-M-07: Оформлення видаткової накладної
Опис: Менеджер або Адміністратор може вручну створити видаткову накладну для списання товарів зі складу з причин, не пов'язаних із замовленнями (пошкоджений товар, втрата, повернення постачальнику, інвентаризація). При поверненні товару постачальнику (причина = RETURN) обов'язково вказується фірма-постачальник, якій повертається товар. Для інших причин списання фірма не вказується. При збереженні накладної система автоматично зменшує поточний залишок товару на складі. Всі операції логуються для аудиту. Видаткові накладні також створюються автоматично системою при оформленні замовлення користувачем (див. FR-C-02). При автоматичному створенні (Reason = ORDER):
•	companyId = NULL (не вказується фірма)
•	Система копіює інформацію про знижки з OrderItems:
o	originalPrice - базова ціна товару
o	appliedPromotionId - ID застосованої знижки
o	discountAmount - сума знижки в грн
o	finalPrice - фінальна ціна після знижки
Ця інформація критично важлива для:
•	Фінансової звітності (розрахунок реального доходу)
•	Аналізу ефективності акцій (ROI по знижках)
•	Історії складських операцій (повна прозорість руху товарів)
Для ручного створення накладних (Reason = RETURN/DAMAGED/LOST/INVENTORY) поля знижок не заповнюються (NULL).
Ризик: Середній (Ризик помилкового списання товарів або неправильного вибору фірми при поверненні, що призведе до некоректних залишків на складі та помилок в обліку повернень постачальникам. Ризик втрати інформації про знижки при автоматичному створенні.)

Менеджер може фільтрувати фірми за статусом (активні/деактивовані) та здійснювати пошук за назвою. Деактивація фірми є м'яким видаленням - фірма залишається в системі для історії, але не доступна для нових накладних.

5. Архітектура та технології системи
5.1. Бізнес-модель
Інтернет-магазин сувенірної продукції Херсонського державного університету побудований за моделлю нішевого інтернет-магазину (Niche E-shop). Ця бізнес-модель орієнтована на вузьку цільову аудиторію - студентів, викладачів, випускників і гостей університету, які зацікавлені у придбанні брендованої сувенірної продукції. Асортимент магазину обмежений товарами з символікою університету (одяг, канцелярія, аксесуари), а цінність полягає у створенні корпоративного іміджу та підтриманні студентської ідентичності. Таким чином, проєкт реалізує вузьку нішу, але з усіма компонентами класичного інтернет-магазину (каталог, кошик, оплата, доставка, облік товарів, адміністрування).
Взаємодія ролей користувачів з даними:
Кожна роль користувача в системі має специфічні права доступу до даних та унікальні інтерфейси для взаємодії з ними. Нижче наведено детальний опис взаємодії кожної ролі з даними системи.
1. Гість (Guest)
Характеристика: Неавторизований відвідувач сайту з обмеженим доступом до функціоналу.
Дані, з якими взаємодіє:
•	Читання (Read): 
o	Products (товари): назва, опис, ціна, вага, зображення
o	Categories (категорії): назва, ієрархія
o	ProductImages (зображення товарів)
o	Публічна інформація про наявність товару (без точних залишків)
Інтерфейси взаємодії:
•	Головна сторінка: каталог товарів з можливістю сортування та попереднього перегляду
•	Сторінка категорій: перегляд товарів за категоріями
•	Сторінка товару: детальна інформація про товар (фото, опис, ціна, відгуки)
•	Пошук та фільтри: пошукова панель, фільтри за категоріями, ціною, розміром
•	Форма реєстрації: поля для створення облікового запису
•	Форма авторизації: поля для входу в систему
Обмеження:
•	Не може додавати товари до кошика
•	Не може оформлювати замовлення
•	Не має доступу до особистого кабінету
•	Не бачить історії замовлень
2. Користувач (Customer)
Характеристика: Зареєстрований клієнт магазину з повним доступом до покупок.
Дані, з якими взаємодіє:
•	Читання (Read): 
o	Products, Categories, ProductImages (як Гість)
o	Users (власний профіль): FirstName, LastName, Email, Phone,
o	studentStatus, gpa, studentVerifiedAt, studentExpiresAt, CreatedAt
o	Cart, CartItems (власний кошик): товари, кількість
o	Orders, OrderItems (власні замовлення): номер, статус, дата,
o	сума, товари, originalPrice, appliedPromotionId, discountAmount, finalPrice
o	Shipping (власні адреси доставки): населений пункт, відділення
o	Nova Poshta
o	Payments (власні платежі): статус оплати, метод, сума
o	Promotions (доступні знижки): назва, опис, тип, значення, термін дії, умови
o	UserPromotions (персональні знижки): assignedAt, usedCount
•	Запис (Write): 
o	Cart, CartItems: додавання/видалення товарів, зміна кількості
o	Users (власний профіль): редагування контактних даних, зміна паролю
o	Shipping: збереження нових адрес доставки
o	Orders: створення нових замовлень

•	Особистий кабінет: 
o	Вкладка "Профіль": 
	Редагування ім'я, прізвища, email, телефону, паролю
	Перегляд студентського статусу (якщо є): GPA, термін дії знижок
	Кнопка "Верифікувати студентський статус" (якщо не верифіковано)

	Для кожного замовлення: деталізація цін зі знижками
o	Вкладка "Адреси доставки": 
	Список збережених адрес з можливістю додавання/видалення
o	Вкладка "Мої знижки": 
	Список всіх доступних знижок (студентські, персональні, акційні)
	Для кожної знижки: назва, розмір, область застосування, термін дії
	Історія використаних знижок з фільтрами за датою
•	Сторінка кошика: 
o	Список товарів з мініатюрами, назвами, цінами

o	Розділ "Доставка": вибір населеного пункту та відділення Nova Poshta (випадаючі списки з пошуком)
o	Розділ "Знижки":
	Відображення автоматично застосованих знижок (студентські, персональні, акційні)
	Поле для введення промокоду
	Кнопка "Застосувати промокод"
	Деталізація знижок після застосування промокоду
o	Розділ "Спосіб оплати": радіо-кнопки (Карткою онлайн / При отриманні)
o	Відображення розрахунку вартості доставки
o	Підсумкова сума замовлення з деталізацією:
	Вартість товарів (без знижок)
	Застосовані знижки (деталізація по кожній)
	Проміжна сума товарів

o	Кнопка "Підтвердити замовлення"

o	Дерево категорій з можливістю drag-and-drop для зміни ієрархії

o	Вкладка "Поточні залишки":

o	IntegrationSettings (налаштування інтеграцій): API ключі, webhook URLs
o	BackupRecords (записи резервних копій): дата, розмір, статус
o	SecuritySettings (налаштування безпеки): JWT параметри, CORS політики

o	Permissions: призначення дозволів ролям
o	Users: призначення ролей користувачам
o	IntegrationSettings: налаштування Stripe (API ключі, webhook), Nova Poshta (API ключ), University API (API ключ, Base URL, параметри верифікації)
o	SystemLogs: очищення старих логів
o	BackupRecords: ініціювання резервного копіювання, відновлення з бекапу
o	SecuritySettings: налаштування JWT, CORS, HTTPS сертифікатів

5.2. Загальна архітектура
Інформаційна система інтернет-магазину сувенірної продукції Херсонського державного університету побудована за клієнт-серверною трирівневою архітектурою.
Система складається з таких рівнів:
•	Frontend (клієнтська частина) - вебінтерфейс для користувачів (React), який відповідає за відображення товарів, оформлення замовлень, взаємодію з кошиком та перегляд статусу доставки.
•	Backend (серверна частина) - API, реалізоване на платформі .NET Core (C#), яке обробляє запити, керує бізнес-логікою, взаємодіє з базою даних і зовнішніми сервісами.
•	Database (база даних) - MySQL, у якій зберігаються дані про користувачів, товари, замовлення, платежі, склади та фінансові операції.
Архітектура передбачає REST API для взаємодії між фронтендом і бекендом, а також підтримує інтеграції зі сторонніми сервісами - платіжними шлюзами (Stripe Sandbox) і логістичними компаніями (Nova Poshta API).
5.3. Структура бази даних
Система інтернет-магазину сувенірної продукції Херсонського державного університету базується на гібридній структурі бази даних (SQL + NoSQL). Основна частина даних (користувачі, товари, замовлення, оплати, доставка) зберігається у реляційній базі даних MySQL, яка забезпечує чіткі зв’язки між сутностями та підтримує транзакційність. Допоміжні дані (логи, кеш, аналітика) зберігаються у NoSQL-сховищі, що забезпечує високу швидкодію при обробці неструктурованої інформації. Така архітектура дозволяє поєднати стабільність і надійність реляційної моделі з гнучкістю та продуктивністю нереляційного зберігання.
5.4. Модулі системи:
	1. CatalogModule (Модуль каталогу товарів)
Відповідає за весь функціонал, пов'язаний з товарами та категоріями. Зокрема: перегляд каталогу, пошук і фільтрація товарів, перегляд детальної інформації про товар, а також управління каталогом, категоріями, зображеннями, акціями та запасами на складі з боку менеджера.
	2. CartModule (Модуль кошика)
Відповідає за тимчасове зберігання товарів, вибраних користувачем для майбутнього замовлення.
	3. OrderModule (Модуль замовлень)
Керує повним життєвим циклом замовлення: створення, зберігання, перегляд історії замовлень та відстеження їх статусів.
	4. PaymentModule (Модуль платежів)
Забезпечує процес оплати замовлень через два способи: онлайн-оплата через інтеграцію з Stripe (з підтримкою мультивалютності через Adaptive Pricing) та оплата при отриманні. Модуль відстежує статуси платежів, обробляє webhook від Stripe, зберігає історію транзакцій та підтримує різні методи оплати. Всі суми зберігаються в базовій валюті UAH, конвертація для міжнародних клієнтів відбувається автоматично на стороні Stripe.	
5. UserModule (Модуль користувачів)
Відповідає за реєстрацію, автентифікацію, авторизацію (RBAC), керування особистим кабінетом користувача та його профілем.
	6. AdminPanelModule (Адміністративний модуль)
Надає об'єднаний інтерфейс для адміністраторів та менеджерів для керування всіма аспектами роботи магазину: замовленнями, користувачами, фінансовими звітами та системними налаштуваннями.
	7. ShippingModule (Модуль доставки)
Відповідає за всі процеси, пов'язані з доставкою: інтеграція з API служби доставки Nova Poshta для отримання списку населених пунктів та відділень, керування адресами доставки, автоматичний розрахунок вартості доставки на основі ваги товарів та відстані, відстеження посилок. Модуль забезпечує зручний вибір пункту доставки через інтерактивні списки з пошуком.
	8. NotificationModule (Модуль сповіщень)
Забезпечує відправку сповіщень користувачам, зокрема email-сповіщення про зміну статусу замовлення та персональні пропозиції.
9. WarehouseModule (Модуль складського обліку)
Відповідає за документування руху товарів через прибуткові та видаткові накладні, а також за керування фірмами-постачальниками. Забезпечує автоматичний розрахунок поточних залишків на складі на основі всіх документів та ведення повної історії операцій з товарами. Модуль зберігає інформацію про фірми-постачальників (назва, контакти, адреса) та прив'язує їх до прибуткових накладних для відстеження джерел надходження товарів. При поверненні товарів постачальникам система фіксує в видатковій накладні, якій саме фірмі здійснюється повернення. Модуль інтегрується з OrderModule для автоматичного створення видаткових накладних при оформленні замовлень та з CatalogModule для відображення актуальних залишків товарів.
10. PromotionModule (Модуль акцій та знижок)
Відповідає за повний життєвий цикл акцій та знижок: створення, редагування, деактивацію, видалення. Забезпечує розрахунок фінальних цін з урахуванням знижок, валідацію промокодів, стакування знижок за пріоритетом. Модуль підтримує різні типи знижок (відсоткові, фіксовані, спеціальна ціна), цільові аудиторії (всі користувачі, студенти, персональні), області застосування (товар, категорія, кошик, доставка). Автоматично призначає студентські знижки при верифікації через UniversityModule. Зберігає історію використання знижок та надає аналітику ефективності акцій (кількість використань, загальна сума знижок, ROI). Інтегрується з OrderModule для застосування знижок при оформленні замовлень, з WarehouseModule для збереження інформації про знижки в видаткових накладних, з UserModule для роботи з персональними знижками.
11. UniversityModule (Модуль інтеграції з університетом)
Відповідає за взаємодію з University API для автоматичної верифікації студентського статусу. При реєстрації користувача з університетським email (@ksu.edu.ua або @student.ksu.edu.ua) модуль відправляє запит до University API для отримання даних: studentStatus (REGULAR/SCHOLARSHIP/ HIGH_ACHIEVER), GPA, scholarshipStatus. На основі цих даних автоматично призначаються студентські знижки через PromotionModule. Модуль керує терміном дії studentStatus (за замовчуванням 4 місяці до кінця семестру) та створює queue для повторної верифікації при недоступності API. Забезпечує логування всіх запитів до University API для аудиту. Інтегрується з UserModule для оновлення даних користувачів та з PromotionModule для призначення знижок.
12. AnalyticsModule (Модуль аналітики)
Відповідає за збір, обробку та візуалізацію аналітичних даних для прийняття управлінських рішень. Модуль агрегує дані з Orders, OrderItems, Products, Categories, Promotions, IncomingDocuments, OutgoingDocuments для розрахунку ключових метрик ефективності магазину. Забезпечує розрахунок KPI (загальна кількість замовлень, дохід, середній чек, конверсія), аналіз популярності товарів та категорій, оцінку ефективності акцій та знижок, фінансовий аналіз (витрати, прибуток, ROI), аналіз поведінки користувачів. Модуль підтримує кешування розрахованих метрик для пришвидшення генерації звітів, візуалізацію даних у вигляді графіків та таблиць (line charts, pie charts, bar charts), експорт звітів у форматах PDF, CSV, Excel, фільтрацію та групування даних за періодами (день/тиждень/місяць). Інтегрується з OrderModule, ProductModule, PromotionModule для отримання даних, з WarehouseModule для фінансових розрахунків (витрати на закупівлю), з UserModule для аналізу поведінки користувачів, з ReportService для генерації експортованих файлів.
