# Структура бази даних інтернет-магазину
1. Реляційна структура (SQL) - дані зберігаються в таблицях, які мають чітко визначені стовпці та типи даних. Таблиці пов'язані між собою ключами: первинними (PK) та зовнішніми (FK). Наприклад, таблиця Orders пов'язана з Users через user_id і з Order_Items через order_id.  
   Плюси: чітка структура і зв’язки між даними; зручно робити фінансовий облік та звіти; підтримує транзакції, що гарантує цілісність даних (ACID).  
   Мінуси: складніше масштабувати горизонтально при великих обсягах даних; зміни структури таблиць (додавання/зміна полів) вимагають міграцій.  
   Приклади СУБД: PostgreSQL, MySQL, MS SQL Server, Oracle DB.  
   Використання: класичні інтернет-магазини, облік замовлень, фінансів, користувачів (Amazon, Rozetka, Prom.ua).  
2. Документно-орієнтована структура (NoSQL) - дані зберігаються у вигляді документів (JSON, BSON), які можуть містити вкладені структури та масиви. Наприклад, один документ Order може містити всю інформацію про користувача, товари, оплату та доставку.  
   Плюси: гнучка схема - можна додавати нові поля без змін структури; легка горизонтальна масштабованість; швидкий доступ до даних.  
   Мінуси: складно реалізовувати строгі зв’язки між сутностями; фінансовий облік і транзакції менш надійні; аналітика складніша, ніж у SQL.  
   Приклади СУБД: MongoDB, CouchDB, Firebase.  
   Використання: каталоги товарів з динамічними атрибутами, історія активностей користувача, аналітика великих обсягів даних.  
3. Гібридна структура (SQL + NoSQL) - комбінує переваги обох підходів. Реляційна база зберігає критично важливі дані (користувачі, фінанси, замовлення), а NoSQL - гнучкі дані (логи, перегляди, аналітика, кешовані документи).   
   Плюси: оптимізує різні завдання під різні типи даних; можна швидко обробляти великі обсяги даних для аналітики.  
   Мінуси: складніше адмініструвати; потрібно підтримувати дві системи; синхронізація даних може бути додатковою проблемою.  
   Приклади СУБД: PostgreSQL + MongoDB, MySQL + Redis.  
   Використання: великі маркетплейси з активною аналітикою та рекомендаційними системами.  
4. Графова структура - дані представлені у вигляді вузлів і зв’язків між ними. Кожен вузол - сутність (користувач, товар, замовлення), а ребра показують відносини (купив, переглянув, рекомендовано).  
   Плюси: ідеально для побудови рекомендаційних систем, соціальних зв’язків, пошуку схожих товарів; дозволяє швидко обчислювати шляхи та зв’язки.  
   Мінуси: не дуже зручно для класичного обліку замовлень і фінансів; аналітика і звіти менш очевидні; зв’язки можуть ускладнювати базові CRUD-операції.  
   Приклади СУБД: Neo4j, ArangoDB, OrientDB.  
   Використання: рекомендаційні системи, маркетингові інструменти для підбору товарів, аналіз поведінки користувачів.
## Бажаний варіант:
Для класичного інтернет-магазину (Retail / e-shop) пріоритет - чітка структура і зв’язки між таблицями: користувачі, товари, замовлення, оплата, доставка.  
Реляційна база даних (SQL) + NoSQL оптимальна, оскільки:  
  - Підтримує фінансовий облік і правильний облік товарів.  
  - Легко реалізуються CRUD-операції.  
  - Добре підходить для побудови ER-діаграм для курсової.  
  - Дозволяє показати знання зв’язків 1:1, 1:М, М:N.
## Обраний варіант:
Реляційна база даних (SQL) + NoSQL
