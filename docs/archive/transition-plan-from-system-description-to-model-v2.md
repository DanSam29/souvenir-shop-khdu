# Як здійснити перехід від опису системи до моделі?
Опис системи - це текстове/словесне пояснення, що “що має робити система, з яких модулів, які зв’язки”.  
Модель - це формалізоване представлення цієї системи (діаграми: компоненти, класи, колаборації, потоки, структури).  

| Крок | Суть | Результат |
|------|------|-----------|
| 1. Збір вимог / функціональні сценарії ✅ | Початковий етап, під час якого визначаються мета проєкту, користувачі, ролі, функціональні та нефункціональні вимоги. Цей етап створює базу для подальшого моделювання - саме з цих вимог далі формуються модулі, сценарії взаємодії та архітектура системи. | Документ з усіма вимогами; Діаграма вимог (Requirements Diagram). |
| 2. Моделювання сценаріїв використання ✅ | На основі функціональних вимог створюються діаграми прецедентів (Use Case) для кожної ролі користувача. Визначаються основні сценарії взаємодії користувачів із системою, їх цілі та очікувані результати. Це допомагає зрозуміти, які функції система має надавати та як користувачі будуть з нею працювати. | Use Case Diagram для кожної ролі користувачів. |
| 3. Моделювання бізнес-процесів ✅ | Використовуючи визначені прецеденти, деталізуються ключові бізнес-процеси системи за допомогою діаграм діяльності (Activity Diagram). Описується послідовність дій, розгалуження, паралельні процеси для таких операцій як оформлення замовлення, обробка платежу, доставка товару. Це створює чітке розуміння логіки роботи системи. | Activity Diagram для ключових процесів. |
| 4. Визначення модулів та їх взаємодії ✅ | Спираючись на вимоги та бізнес-процеси, система розділяється на логічні модулі (CatalogModule, CartModule, OrderModule та ін.). Визначаються зв'язки між модулями: які дані або дії один модуль передає іншому, які інтерфейси необхідні для взаємодії. Це встановлює структуру системи та її основні блоки. | Package Diagram (структура модулів); Component Diagram (компоненти та їх інтерфейси); Таблиця інтерфейсів (опис REST endpoints / подій / структур даних між модулями). |
| 5. Побудова архітектурної моделі ✅ | На основі визначених модулів і зв'язків між ними створюється багаторівнева архітектура системи. Використовується C4 Model для відображення контексту системи (рівень C1), контейнерів - Frontend, Backend, БД (рівень C2), та компонентів всередині контейнерів (рівень C3). Визначається розгортання системи та взаємодія з зовнішніми сервісами (Stripe, Nova Poshta). | C4 Model діаграми (C1: System Context, C2: Container, C3: Component); Deployment Diagram (схема розгортання на Render.com з Docker); Опис архітектурних рішень та технологічного стеку. |
| 6. Деталізація структури даних ✅ | Використовуючи архітектуру та вимоги до даних, деталізується внутрішня структура системи. Створюються діаграми класів для кожного модуля з атрибутами, методами та зв'язками. Проектується структура бази даних (ER-діаграма) з усіма таблицями, полями, зв'язками та обмеженнями. На цьому етапі застосовуються вимоги до бізнес-логіки та зберігання даних. | Class Diagram для кожного модуля (з атрибутами, методами, зв'язками); ER-діаграма бази даних MySQL (таблиці, поля, зв'язки, ключі); Опис структур даних (DTO, entities). |
| 7. Моделювання динамічної поведінки ✅ | Для критичних сценаріїв, що були раніше описані на діаграмах діяльності, створюються діаграми послідовності (Sequence Diagram), що показують взаємодію між об'єктами в часі: виклики методів, передачу повідомлень, порядок операцій. Для об'єктів зі складною поведінкою створюються діаграми станів (State Machine Diagram), наприклад, для замовлення (Створено → Оплачено → Відправлено → Доставлено). | Sequence Diagram для ключових сценаріїв; State Machine Diagram. |
| 8. Верифікація та трасування вимог ✅ | Використовуючи всі побудовані моделі, проводиться перевірка повноти та коректності: чи кожна функціональна та нефункціональна вимога має своє представлення у вигляді модуля, класу, прецеденту або сценарію. Створюється матриця трасування, яка пов'язує вимоги з елементами моделей. Це дозволяє переконатись, що система повністю покриває всі поставлені цілі. | Requirements Traceability Matrix (зв'язок вимог з Use Cases, модулями, класами). |
| 9. Формування технічної документації ✅ | На завершення всі створені моделі та діаграми об'єднуються в єдиний документ System Model Specification. Додається опис архітектурних рішень, обґрунтування вибору технологій, інструкції щодо розгортання та інтеграції. Цей етап завершує фазу проектування і готує проєкт до безпосередньої розробки (імплементації). | System Model Specification (SRS + всі діаграми). |
