# Як здійснити перехід від опису системи до моделі?
Опис системи - це текстове/словесне пояснення, що “що має робити система, з яких модулів, які зв’язки”.  
Модель - це формалізоване представлення цієї системи (діаграми: компоненти, класи, колаборації, потоки, структури).  

| Крок | Суть | Результат |
|------|------|-----------|
| 1. Збір вимог / функціональні сценарії | Опис: хто користувач, що він може робити (реєстрація, перегляд товарів, добавлення в кошик, оплата, історія). | Use-case або user story список. |
| 2. Визначення модулів / підсистем|Виходячи з функцій, розбити систему на модулі.|Список модулів із їх межами відповідальностей. |
| 3. Визначення інтерфейсів між модулями | Для кожного модуля описати, які API він надає/використовує, якими даними спілкується. | Контракти (REST endpoints / події / передані структури). |
| 4. Побудова архітектурної моделі | Візуалізувати: компонентна діаграма (як ми робили), контейнери (C4 model), взаємодії. | UML / C4 діаграми, які показують, як модулі зв’язані. |
| 5. Деталізація моделі | Перейти від компонентної до класової моделі (клас-діаграми), моделі даних (ER-діаграми). | Класи, атрибути, методи, зв’язки; окремо ER-діаграма бази даних. |
| 6. Верифікація (перевірка) моделі з вимогами | Перевірити, чи всі функціональні сценарії покриті моделями. | Відповідність use-case → модулі → класи. |
| 7. Генерація технічних діаграм / документації | Додати sequence diagrams (для сценаріїв), activity diagrams, deployment diagrams. | Повна модель системи, готова до реалізації. |
