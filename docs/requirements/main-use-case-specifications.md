# Специфікації основних прецедентів
## UC-G-01: Реєстрація користувача в системі
**Загальна інформація**  
  - Актор: Гість  
  - Пріоритет: Високий  
  - Ризик: Середній (витік або компрометація облікових даних)  
  - Складність: Середня  
  - Залежності: FR-G-04, University API (для верифікації студентів)  

**Опис**  
  Гість створює новий обліковий запис в системі для отримання доступу до функцій Користувача (оформлення замовлень, перегляд історії, особистий кабінет). Система автоматично перевіряє університетський email та призначає студентські знижки відповідно до статусу студента.  

**Передумови**  
  - Гість знаходиться на сайті інтернет-магазину
  - Гість не авторизований в системі
  - Email адреса гостя ще не зареєстрована в системі

**Постумови (успішний сценарій)**
  - Новий обліковий запис створено в базі даних
  - Користувачу призначена роль "Користувач"
  - Користувач автоматично авторизований в системі
  - Email-підтвердження надіслано на вказану адресу
  - Створено порожній особистий кабінет
  - Якщо email університетський - перевірено статус студента та призначено відповідні знижки

**Основний сценарій (Success Path)**
1. Гість натискає кнопку "Реєстрація" в інтерфейсі
2. Система відображає форму реєстрації з полями:
    - Ім'я (обов'язкове)
    - Прізвище (обов'язкове)
    - Email (обов'язкове)
    - Пароль (обов'язкове, мін. 8 символів)
    - Підтвердження паролю (обов'язкове)
    - Телефон (опціонально)
    - Згода з умовами використання (обов'язкове, checkbox)
3. Гість заповнює всі обов'язкові поля
4. Гість встановлює прапорець "Згода з умовами використання"
5. Гість натискає кнопку "Зареєструватися"
6. Система валідує введені дані на клієнті (формат, довжина, співпадіння паролів)
7. Система надсилає запит на Backend API
8. Backend перевіряє унікальність Email в базі даних
9. Backend хешує пароль за допомогою bcrypt
10. Backend створює новий запис користувача в таблиці Users з полями: FirstName, LastName, Email, PasswordHash, Phone, CreatedAt
11. Backend призначає роль "Користувач"
12. Backend створює порожній особистий кабінет
13. Backend перевіряє, чи email належить до домену університету (@ksu.edu.ua)
14. Якщо email університетський:
    - Backend надсилає запит до University API для верифікації студента (GET /students/{email})
    - Якщо студент знайдений, Backend отримує дані: studentStatus, GPA, scholarshipStatus
    - Backend оновлює запис User: studentStatus, gpa, studentVerifiedAt = now(), studentExpiresAt = now() + 4 місяці
    - Система автоматично призначає студентські знижки залежно від studentStatus (REGULAR - базова, SCHOLARSHIP - підвищена, HIGH\_ACHIEVER (GPA ≥ 4.5) - максимальна)
    - Інформація про знижки додається в Email-підтвердження
15. Якщо email не університетський або студент не знайдений - встановити studentStatus = NONE
16. Backend генерує JWT токен для автоматичної авторизації
17. Backend підготовує Email з підтвердженням реєстрації (включно з інформацією про знижки, якщо є)
18. Backend надсилає Email через Email сервіс
19. Backend логує успішну реєстрацію в Serilog
20. Backend повертає успішну відповідь з JWT токеном
21. Frontend отримує JWT токен
22. Frontend зберігає JWT токен в пам'яті
23. Система перенаправляє користувача на головну сторінку
24. Система відображає повідомлення "Реєстрація успішна! Перевірте Email для підтвердження"

      + Умова: Checkbox "Згода з умовами" не встановлено

          3. Use case завершується невдало  

**Бізнес-правила**  
1. BR-1: Email має бути унікальним в системі
2. BR-2: Пароль має містити мінімум 8 символів
3. BR-3: Пароль має містити хоча б одну букву та одну цифру
4. BR-4: Обов'язкові поля: Ім'я, Прізвище, Email, Пароль, Згода з умовами
5. BR-5: За замовчуванням новому користувачу призначається роль "Користувач"
6. BR-6: Паролі зберігаються тільки в хешованому вигляді (bcrypt)
7. BR-7: Email домену @ksu.edu.ua автоматично перевіряється через University API
8. BR-8: Студентські знижки призначаються автоматично на основі studentStatus
9. BR-9: Студентський статус має термін дії 4 місяці (один семестр)

**Нефункціональні вимоги**
  - NFR-P-02: Час обробки реєстрації не повинен перевищувати 2 секунди
  - NFR-S-02: Використання JWT для аутентифікації
  - NFR-S-04: Захист від XSS та CSRF атак
  - NFR-U-01: Форма реєстрації має бути інтуїтивно зрозумілою
  - NFR-U-02: Форма реєстрації має коректно відображатися на різних пристроях (ПК, планшети, смартфони)

**Частота використання**  
  Висока (кожен новий клієнт магазину)  

**Технічні примітки**
  - Використовується ASP.NET Identity для керування користувачами
  - JWT токени мають термін дії 24 години
  - Email підтвердження не є обов'язковим для використання системи, але рекомендується
  - University API викликається асинхронно, тому недоступність API не блокує реєстрацію
## UC-C-01: Оформлення замовлення
**Загальна інформація**
  - Актор: Користувач
  - Пріоритет: Високий
  - Ризик: Високий (втрата даних замовлення, помилки інтеграції)
  - Складність: Висока
  - Залежності: FR-C-01, FR-C-02 зовнішні API (Nova Poshta, Stripe)

**Опис**  
  Користувач оформляє замовлення з товарів у кошику, система автоматично розраховує знижки (студентські, персональні, акційні), користувач може застосувати промокод, вибирає спосіб доставки Nova Poshta, пункт відправлення та здійснює оплату через Stripe або обирає оплату при отриманні.

**Передумови**
  - Користувач авторизований в системі (роль: Користувач)
  - У кошику є хоча б один товар
  - Всі товари в кошику доступні на складі
  - Загальна сума замовлення >= 100 грн
  - Інтеграції Nova Poshta API та Stripe API налаштовані та працюють

**Постумови (успішний сценарій)**
  - Замовлення створено в базі даних
  - Статус замовлення встановлено ("Обробляється" для оплати онлайн або "Очікує оплати при отриманні")
  - Створено OrderItems з повною інформацією про знижки (originalPrice, appliedPromotionId, discountAmount, finalPrice)
  - Автоматично створено видаткові накладні через WarehouseService
  - Товари зарезервовано на складі (зменшено Stock)
  - Для онлайн оплати - платіж оброблено через Stripe
  - Дані доставки Nova Poshta збережено
  - Email-підтвердження надіслано користувачу з деталізацією знижок
  - Кошик очищено
  - Якщо використано промокод - оновлено Promotions.currentUsage

**Основний сценарій (Success Path)**
1. Користувач переходить на сторінку кошика
2. Система відображає список товарів у кошику з загальною сумою
3. Система перевіряє доступні знижки для користувача: студентські знижки (якщо studentStatus активний), персональні знижки (UserPromotions), акційні знижки (активні Promotions)
4. Система попередньо розраховує суму з урахуванням автоматичних знижок
5. Система відображає деталізацію: Сума товарів, Автоматичні знижки, Проміжна сума
6. Користувач натискає кнопку "Оформити замовлення"
7. Система перевіряє наявність всіх товарів на складі через БД
8. Система перевіряє мінімальну суму замовлення (100 грн)
9. Система відображає форму оформлення замовлення
10. Користувач перевіряє/редагує контактні дані (ім'я, телефон, email - автозаповнення з профілю)
11. Користувач вибирає спосіб доставки "Nova Poshta"
12. Система надсилає запит до Nova Poshta API (список населених пунктів)
13. Система відображає список населених пунктів з пошуком
14. Користувач вибирає населений пункт
15. Система надсилає запит до Nova Poshta API (список відділень у населеному пункті)
16. Система відображає список відділень Nova Poshta
17. Користувач вибирає відділення Nova Poshta
18. Система надсилає запит Nova Poshta API (розрахунок вартості доставки)
19. Система отримує вартість доставки з API
20. Система розраховує загальну суму (товари + доставка)
21. Система відображає загальну суму з деталізацією: Вартість товарів (без знижок), Застосовані знижки, Проміжна сума товарів, Вартість доставки, Підсумок до оплати
22. Система відображає поле для введення промокоду (опціонально)
23. Система відображає вибір способу оплати: Карткою онлайн, Оплата при отриманні
24. Якщо користувач бажає ввести промокод:
    1) Користувач вводить промокод
    2) Backend (PromotionService) отримує промокод
    3) Backend перевіряє існування промокоду в БД
    4) Backend перевіряє валідність: isActive = true, startDate <= now <= endDate, currentUsage < usageLimit
    5) Backend перевіряє, чи промокод вже застосований до цього замовлення
    6) Backend застосовує промокод та розраховує додаткову знижку
    7) Backend перераховує загальну суму з урахуванням промокоду
    8) Система відображає оновлену деталізацію з промокодом
    9) Система показує "Промокод успішно застосовано!"
25. Користувач обирає спосіб оплати:  
  Для оплати карткою онлайн:  
26. Користувач натискає "Підтвердити замовлення"
27. Backend (OrderService) отримує дані замовлення з Frontend
28. Backend (PromotionService) розраховує фінальні ціни для кожного товару: originalPrice (базова ціна), appliedPromotionId (ID найвигіднішої знижки або промокоду), discountAmount (сума знижки), finalPrice (ціна після знижки)
29. Backend створює запис замовлення в БД (status: "Очікує оплати")
30. Backend створює OrderItems з повною інформацією про знижки (productId, quantity, originalPrice, appliedPromotionId, discountAmount, finalPrice)
31. Backend створює видаткові накладні через WarehouseService (автоматично зменшує Stock)
32. Backend генерує сесію оплати в Stripe через API
33. Система перенаправляє користувача на сторінку Stripe
34. Користувач вводить дані банківської картки
35. Stripe обробляє платіж
36. Stripe повертає статус успішної оплати через webhook
37. Backend отримує webhook від Stripe
38. Backend перевіряє підпис webhook (webhook signature)
39. Backend оновлює статус замовлення на "Обробляється"
40. Backend оновлює статус платежу на "Оплачено"
41. Backend додає запис у таблицю Payments
42. Backend додає запис у таблицю OrderHistory
43. Backend надсилає Email-підтвердження користувачу з деталізацією: Номер замовлення, Список товарів, Вартість товарів (без знижок), Застосовані знижки (деталізація по кожній), Вартість доставки, Підсумок оплачено
44. Backend очищає кошик користувача
45. Система перенаправляє на сторінку "Замовлення успішно створено"
46. Система відображає номер та деталі замовлення  
  Для оплати при отриманні:

**Передумови**

## UC-M-02: Оформлення прибуткової накладної

Менеджер оформляє прибуткову накладну для фіксації надходження товару на склад від постачальника. Система автоматично оновлює залишки на складі (Stock).  

**Передумови**
  - Менеджер авторизований в системі (роль: Менеджер або Адміністратор)

  - Список фірм-постачальників створений у системі (таблиця Companies)

  - Створено запис в таблиці IncomingDocuments з даними про надходження

**Основний сценарій (Success Path)**
1. Менеджер входить в адміністративну панель

**Бізнес-правила**
1. BR-1: API ключі зберігаються в зашифрованому вигляді
2. BR-2: Secret Key та Webhook Secret відображаються замасковані в UI
3. BR-3: Для розробки використовується Sandbox режим з тестовими ключами (pk\_test, sk\_test)
4. BR-4: Для Production використовуються реальні ключі (pk\_live, sk\_live)

6. BR-6: Зміна API ключів вимагає повторного тестування

  - NFR-S-04: Webhook signature має верифікуватися для безпеки
  - NFR-R-03: Система має коректно обробляти помилки Stripe API
